2) KB rules phase — Boolean expressions only in `if` and `then`
- Each item in `if` / `then` lists must be a Boolean expression, using only:
  - Atom: {"pred":"Name","args":[...],"negated":false}  (alias key "symbol" instead of "pred"; optional "neg" instead of "negated")

    - This is different from {"not": expr}, which negates an arbitrary Boolean subexpression.
  - {"and":[expr,...]}, {"or":[expr,...]}, {"not": expr}
  - Compare: {"compare":{"left":TERM,"op":"<=","right":TERM}} or the same three fields as a flat object {"left":...,"op":"<=","right":...}. Use only supported comparison operators as in the KB compiler.
- TERM (for compare left/right, and for non-Bool-valued positions if any): variable string, number, or {"func":"FunctionName","args":[...]} with arity from the symbol table.
- NEVER put {"func":...} as a standalone list element in `if` or `then` — function calls are not Booleans. For thresholds, use `compare` and/or Bool helper predicates declared in the symbol table.
- Compare sorts: `left` and `right` must have the same scalar sort (Int with Int, Real with Real, Date with Date). Do not compare Int to Real or pass a Date/FinancialYear variable where Int is required unless the function signature uses that sort.
- Quantifier types in `forall` must use type names exactly as declared in `types` / SYMBOL_TABLE (e.g. `FinancialYear`, not `financial_year`). Every variable's `type` must match every predicate/function argument position where that variable appears.
- NEVER use the strings `true` or `false` as predicate names in atoms — they are literals, not predicates.
- Every identifier called as a function or threshold in a rule must be declared under `functions` in Step 1 (e.g. `threshold_employee_small: -> Int`), never only mentioned inside rules.
- Generic `not` example: {"not": {"pred":"p_hold","args":["x"]}}  (outer expr is Bool).
- Generic `compare` example: {"left": {"func":"f_metric","args":["x"]},"op":"<=","right": 100}
