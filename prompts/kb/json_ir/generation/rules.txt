## HIGH-PRIORITY RULE RULES

1. Use the frozen symbol table exactly.
- Every predicate atom must use a predicate from SYMBOL_TABLE.
- Every function term must use a function from SYMBOL_TABLE.
- Never use a function name as a predicate atom.
- Never use a predicate name as a function term.

2. Use variables correctly.
- Every variable used in if/then must be declared in forall.
- The declared type must match every use.
- Do not rely on implicit subtyping.
- If a concept is a role/status/subset, quantify the broader type and add a role/status predicate in the if-condition.
- Do not apply both a deceased-style and a surviving-style unary observable to the same variable. Use separate variables and, when needed, a binary relation predicate linking them.
- Do not use placeholder names starting with `_`; every identifier must be declared in forall.

3. Express legal consequences.
- A good rule derives a derived legal result from observable facts/functions and helper conditions.
- Rules that state when consequences apply, take effect, or start from a period must conclude on a derived legal-effect predicate (not only a static classification predicate).
- Definition-style provisions must conclude on a derived predicate representing the legal status/classification under the provision.
- Do not create P(x) => P(x).
- Do not create rules where a derived predicate depends only on itself.
- Do not create rules that only derive observable case inputs.
- Do not create rules with empty then.
- Every derived predicate in SYMBOL_TABLE must appear in at least one rule THEN.

4. No case constants.
- Every identifier used as an argument must be a quantified variable unless it is a numeric literal.
- Do not use concrete names, objects, dates, organizations, IDs, or example constants.

5. Thresholds and combinations.
- Use observable functions for raw measurable values.
- Preserve numeric thresholds exactly as in the scoped law text. Do not invent, adjust, round, or add to legal constants.
- FIRST define per-criterion exceeded helpers in THEN from numeric compares (e.g. employees_exceeded(c,fy) from compare(annual_average_employees_fte(c,fy), >, THRESHOLD)).
- THEN define counting helper H in THEN with pairwise conjunctions: (T1 & T2) OR (T1 & T3) OR (T2 & T3).
- Include BOTH qualification AND exclusion for at-most-one criterion laws:
  (1) qualification: NOT H plus other conditions => classification (non-negated in THEN);
  (2) exclusion: H => NOT classification (use "negated": true in THEN — never C = false).
- Safe negative/exclusion rules:
  Do not create a negative classification/legal-output rule by mechanically negating every condition
  of the positive rule. Under open-world semantics, absence of an observable prerequisite is not
  proof of its negation.
  Prefer: disqualifying_condition(x) => not legal_classification(x)
  Avoid: not prerequisite_condition(x) => not legal_classification(x)
  Avoid OR-rules that mix missing prerequisites with true disqualifiers unless every OR branch is
  explicitly a legal exclusion supported by law.
- Never use a function name as a Bool predicate atom; use compare(func(...), op, value) instead.
- Never use `not helper_predicate(x)` when helper_predicate has no defining rule.
- When the law says not more than one criterion is exceeded: NEVER derive a favorable status from `A_exceeded OR B_exceeded OR C_exceeded` alone.
- For “at least two of A, B, C”, use pairwise combinations: (A and B) or (A and C) or (B and C).
- For “at most one of A, B, C”, negate the “at least two” condition.
- Do not count Boolean conditions with arithmetic.
- Use Real literals (e.g. 900000.0) when comparing Real functions; use Int literals for Int functions.

5b. Helper closure (critical):
- Before finishing, list every helper predicate or helper-valued function used in any rule IF-condition.
- Each such symbol MUST also appear in at least one rule THEN (defining it from observable functions/comparisons or simpler helpers).

Prerequisite status/classification closure:
- Do not use a derived status/classification predicate in IF unless it is defined in THEN, or is observable/background in SYMBOL_TABLE.
- If a prerequisite status can be directly asserted by the case and is not the final legal output, reclassify it as observable with directly_observable=true instead of leaving it derived-but-undefined.
- If the scoped law defines the status from subconditions, add THEN rules from observable facts/functions.
- If the law does not define it and it is not a case-supplied background fact, remove it from IF and encode only locally available conditions in the target rule.
- Do not use an observable broader classification/status in IF as a blocking prerequisite for the target classification unless the case can directly assert it or the scoped law defines it.
- If the broader status is not defined in scope and not case-supplied, remove it from IF and use only locally available legal conditions.
- Never compensate for a missing broader prerequisite by adding a negative rule from not broader_status to not target_classification.
- Do not define prerequisite statuses circularly (A from B while B also depends on undefined A).
- Never leave a derived/helper status used in IF without a corresponding THEN definition or observable/background declaration.

- Helper predicates: define in THEN as P(args) or negated P(args) from compares or simpler helpers.
- Helper functions (Real/Int thresholds): define in THEN with compare equality to a law-text literal, e.g.
  {"compare": {"left": {"func": "adjusted_employee_threshold", "args": ["c", "fy"]}, "op": "=", "right": 50}}
- Prefer direct IF comparisons to law literals (annual_average_employees_fte(c,fy) <= 50) unless an adjusted threshold helper already exists in SYMBOL_TABLE and must be closed.
- If helper H is missing a definition, ADD a new rule with H in THEN rather than deleting H from IF or renaming it.
- Example pattern for exceeds_employee_threshold(c, fy): THEN exceeds_employee_threshold(c, fy) when compare(annual_average_employees(c, fy), >, 50).
- Example pattern for more_than_one_criterion_exceeded(c, fy): THEN when at least two single-criterion exceedance helpers hold for the same (c, fy).
- Never use numeric functions as Bool predicate atoms; never assign false/true to predicates as if they were functions.

5c. Existential conditions without witness entities:
- When the law only requires that some related entity exists (affiliation, consortium, subsidiary link), do NOT call a binary relation with the same variable twice (e.g. affiliated_with(c, c)).
- Prefer an observable Boolean on the main subject (e.g. has_affiliated_companies(c)) unless the identity of the related entity is legally required in the conclusion.
- Apply the same pattern for consortium/parent/subsidiary existence checks referenced in threshold consolidation rules.

6. Raw FO is forbidden.
- Every rule must be object-shaped JSON_IR.
- Do not output raw FO strings.

OR:

Comparison:
{"left": {"func": "value_function", "args": ["x"]}, "op": "<=", "right": 10}

Alternative comparison:
{"compare": {"left": {"func": "value_function", "args": ["x"]}, "op": "<=", "right": 10}}

Function term:
{"func": "function_name", "args": ["x"]}

Function terms may appear inside comparisons or where a term is valid. They are not Boolean expressions.

## SELF-CHECK BEFORE RETURNING

Before returning JSON, verify:
- root key is exactly rules;
- every rule has forall, if, then, and operator;
- then is not empty;
- every symbol exists in SYMBOL_TABLE;
- every arity matches;
- every variable is declared;
- every variable has the correct type in every use;
- no case constants appear;
- no raw FO strings appear;
- no circular or tautological rule appears;
- every rule expresses reusable law, not a case-specific answer.

## CRITICAL CONSEQUENT RULE

The `then` part of a rule should normally contain derived predicates.

Observable predicates represent facts supplied by a case. They belong in `if`, not in `then`.

Do not derive raw case-input facts from legal rules.

If a legal consequence must be represented, use a predicate/function whose kind is `derived`.

If no suitable derived symbol exists, the symbol table must be repaired before rules are generated.

Helper predicates may be defined by rules, but they should not be the only final legal output of the KB.

## Example A — Legal role and legal consequence

LAW FRAGMENT:
If a person is the surviving spouse of the deceased, and the deceased leaves descendants, the surviving spouse receives a usufruct over the deceased's share of the family home.

AVAILABLE SYMBOLS:

- Asset

- deceased_leaves_descendants(Person) observable predicate
- is_family_home_of(Asset, Person) observable predicate
- receives_usufruct_over(Person, Asset) derived predicate

EXPECTED RULE PATTERN:
{

    {
      "forall": [
        {"var": "spouse", "type": "Person"},
        {"var": "deceased", "type": "Person"},
        {"var": "home", "type": "Asset"}
      ],
      "if": [
        {
          "and": [
            {"pred": "is_surviving_spouse_of", "args": ["spouse", "deceased"], "negated": false},

      "operator": "implies"
    }

}

Why this is good:
- The legal role is a condition.
- The legal effect is the conclusion.
- No concrete case names appear in the reusable rule.
- The rule is reusable for any spouse, deceased person, and asset.
