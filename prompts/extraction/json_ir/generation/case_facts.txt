## CASE EXTRACTION RULES

Use KB_SCHEMA as the only authority for type names, predicate names, function names, arities, argument order, and kind metadata.
Use SCHEMA_ENVIRONMENT as the operational contract:
- Use only listed entity types.
- Use only listed predicate/function names.
- Case assertions/value_assertions may use only symbols marked assertable.
- Do not invent entity types.

Prefer symbols with kind = observable.

Do not assert symbols with kind = derived unless the case explicitly states that the legal conclusion has already been legally established.

Do not assert symbols with kind = helper unless the case explicitly states that intermediate condition itself and it is not better represented by more basic observable facts or values.

Controlled factual case input (threshold/criterion satisfaction):
- Prefer numeric value_assertions when the case states explicit numbers.
- If the case explicitly states that a threshold/criterion was exceeded or satisfied but gives no numbers, you MAY assert a listed factual case input predicate from SCHEMA_ENVIRONMENT with evidence_text (source phrase from the case).
- Do NOT invent numeric values to satisfy thresholds (e.g. do not set employee count to 51 when the case only says "threshold exceeded").
- Include evidence_text on every factual threshold/criterion assertion.
- Do NOT assert legal-output, legal-consequence, or query-target predicates as case facts.
- If the case only states a legal conclusion without decomposable observables, omit it and allow unknown.

General rule:
- observable symbols are case inputs;
- derived symbols are legal outputs;
- helper symbols are internal reasoning steps.

For numeric values, amounts, counts, durations, percentages, dates, and other measurable values:
- use value_assertions when the schema provides a function;
- do not convert numeric values into vague Boolean facts if a function exists.

For relationships:
- preserve the direction and roles required by the schema;
- do not swap subject/object, actor/target, applicant/authority, right-holder/object, party/counterparty, event/participant, decision/subject, or period/entity roles.

For entities:
- extract every concrete entity needed by the asserted facts or value assertions;
- use stable lowercase snake_case identifiers;
- if the case mentions exactly one entity of a type, still give it a concrete identifier based on the text;
- do not use vague identifiers such as this_entity, this_person, this_object, unknown, someone, something, or entity.

Negation:
- Use negated=true only when the case explicitly states that the fact does not hold.
- Do not use negation to encode absence of evidence.
- If the case simply does not mention a fact, omit it.

Uncertainty:
- If a fact is not explicit or directly implied, omit it.
- If no valid schema symbol exists for a case detail, omit that detail.
- Never invent a symbol.
