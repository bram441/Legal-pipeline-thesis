## QUERY FIELD RULES

kind:
- Use "predicate" for yes/no (mode=boolean) or which-entities (mode=set) questions about a concrete predicate.
- Use "intent" when the question needs propagation, model expansion, ranges, consistency, optimization, relevance, or explain.
- Public intents only: propagation, model_expansion, get_range, satisfiable, optimization, relevance, explain.
- Never use internal intents deduction or deduction_set (the router derives those from predicate mode).

Intent selection (generic, law-agnostic):
- predicate + boolean: whether a concrete legal status, right, obligation, permission, prohibition, validity result, entitlement, exclusion, classification, or consequence applies.
- predicate + set: which legal actors, objects, events, decisions, or periods satisfy a unary legal condition.
- propagation: what follows; what can be concluded; which legal consequences are certain across all models.
- model_expansion: a possible completed legal situation; one possible assignment or outcome (not certainty).
- get_range: possible values, ranges, amounts, dates, durations, counts, or percentages of a numeric function.
- satisfiable: whether the case and law are jointly consistent.
- optimization: minimum, maximum, earliest, latest, least, greatest, smallest, largest when a numeric objective exists.
- relevance: which symbols or facts matter for the question.
- explain: why, justification, or why not; use target_type=predicate with predicate_hint+args, or target_type=satisfiable for inconsistency.

For kind=intent, fill intent-specific fields from the schema (focus_symbols, focus_entities, max_models, function, direction, target_type).
If unsure between boolean predicate and propagation, prefer propagation when the question asks what is certain, not a single yes/no atom.

predicate_hint:
- Required for kind = "predicate".
- Must be an exact predicate name from KB_SCHEMA.
- Must be Boolean.
- No synonyms.
- No natural-language descriptions.

mode:
- Use "boolean" when the question asks whether a concrete legal result holds.
- Use "set" only when the question asks which entities satisfy a predicate.

args:
- Must match the arity of predicate_hint exactly.
- Must preserve the predicate argument order from KB_SCHEMA.
- Every non-"?" arg must appear verbatim in Normalized case entities.
- Do not invent constants.
- Do not use type names as arguments.
- Do not use placeholders.
- Use "?" only when the query genuinely asks for an unknown entity/set and the contract/schema allows it.

explain:
- true only if the user explicitly asks for explanation, reasons, or why.
- otherwise false.

## QUERY SELECTION RULES

Choose the symbol that directly represents the requested legal answer.

Preferred order:
1. Choose a symbol with kind = derived when the question asks for a legal status, classification, consequence, right, obligation, permission, prohibition, exception, sanction, validity condition, entitlement, exclusion, or legal effect.
2. Choose a symbol with kind = helper only when the question explicitly asks about that intermediate condition.
3. Choose a symbol with kind = observable only when the question asks whether a factual condition from the case is present.

Do not choose an observable input predicate when the question asks for a legal conclusion.
Do not choose a helper predicate when a derived predicate directly answers the question.

For query arguments:
- use only concrete entities already present in Normalized case entities;
- preserve the argument order required by the selected predicate;
- if exactly one entity exists for a required type, use that entity;
- if multiple entities exist, choose the one whose role is identified by the question;
- if no valid entity exists, use "?" only if allowed by the contract and question shape.

Do not introduce new entity names, paraphrased entity names, type names as args, or placeholders.
Query targets must come from query_target_candidate symbols.
Temporal/background support symbols may support reasoning but are not final legal-effect answer predicates when legal-output targets exist.

Classification vs timing selection:
- If the question is of the form "Is X a Y?", "Does X qualify as Y?", "valt X onder Y?", or asks for legal status/classification/category, choose the classification/status predicate that matches Y.
- Do not choose a timing/effect predicate for a classification question, even if that timing/effect predicate is marked legal_output=true.
- Choose timing/effect predicates only when the question asks when something applies, from which period a consequence starts, whether a status change takes effect, or other temporal applicability language.
