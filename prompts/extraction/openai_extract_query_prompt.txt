Rules:
- Output must be VALID JSON.
- Output JSON shape: {"type":"predicate"|"intent","explain":bool,"predicate":"","mode":"set"|"boolean","args":[],"intent":"","symbol":"","entity":""}
- Use lowercase identifiers (snake_case) for constants.
- IMPORTANT: Use ONLY predicate/function symbol NAMES from KB_SCHEMA. Copy EXACTLY (including long names if the KB defines them that way; do not invent new sentence-like symbol names).
- Case and question may be in any language; map them to the schema symbols (schema names are from the KB).
- 'Why X?' / 'Waarom X?' = the SAME predicate as 'Is X?' but with explain=true. Do NOT pick a different predicate.
- For questions about legal conclusions (Is X liable? Is X punishable? Does X qualify?): use the conclusion predicate from KB_SCHEMA that matches. The system infers which rule applies from the case facts.
- CRITICAL: For boolean predicate queries, args MUST be the entity the question asks about. Before returning, check: "Is the question about a specific person?" If yes (e.g. "Is Karel liable?", "What is the sentence for Pieter?"), use THAT person in args or entity. "Is Karel liable?" => args: ["karel"]. "Is Pieter liable?" => args: ["pieter"]. Do NOT use a different person from the case. Do NOT use type names (e.g. Offender, Person) as args—use the concrete person name (e.g. jan, pieter).
- **Predicate arity (read KB_SCHEMA):** For each predicate, `args` in KB_SCHEMA lists domain types in order; that list length is the **arity**. For `type: "predicate"` and `mode: "boolean"`, your `query.args` must contain **exactly that many** strings—one entity identifier per position. Unary predicates (`"args": ["Person"]` only) require `args: ["oneName"]`. **Never** use `args: []` for a unary boolean query. For `mode: "set"`, always use `args: []`.
- **Entity tokens:** Each string in `query.args` must match (lowercase) a constant already used in **case.facts** or listed under **case.entities**. Do not invent generic constants like `person`, `foreigner`, or `alien` unless the case facts literally use that same token as an identifier.
- Role-based wording (no name in the question): If the question refers to a role ("the surviving spouse", "the deceased", "the buyer", …) instead of a name, read the case facts to see which **named** constants play those roles, then put the constant that matches the question in args. Match **who the queried predicate is about** (read KB_SCHEMA arity and informal meaning): e.g. a predicate about "who holds a right" should use the person who holds it in the case, not another party, unless the schema clearly indexes that other party.
- For "Is X liable under Article N?": prefer a predicate that names the article (e.g. punishedArt398, punishedArt399). If the schema has no such predicate but has condition predicates, map by article: Art. 398 = intentional wounds without illness/incapacity; Art. 399 = illness or incapacity to work; Art. 400 = severe/incurable harm; Art. 401 = death without intent to kill. Use the condition predicate that characterizes that article (e.g. CausesIllnessOrIncapacity for Art. 399).
- For questions about quantitative outcomes (amounts, durations, ranges): use type="intent", intent="get_range", symbol=<function name>, entity=<person asked about>. If the question asks "for X" or "about X", set entity to X (lowercase). Pick the function from KB_SCHEMA that matches. Match "minimum" to Min, "maximum" to Max. Prefer a single unit (e.g. Days) when the schema has multiple (Days/Months/Years) for the same quantity.
- Do NOT map to predicates that merely restate facts already given in the case. Map to predicates that require reasoning or rule application.
- Only use type='intent' for: satisfiable, propagation, optimization, get_range.
- For predicate queries: intent="", symbol="".
- For intent get_range: predicate="", mode="set", args=[], symbol=<function>, entity=<person if question specifies one, else "">.
